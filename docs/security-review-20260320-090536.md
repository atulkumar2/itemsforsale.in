# Security Review Report

**Date (UTC):** 2026-03-20 09:05:36

## Scope

Reviewed the authentication flow, public submission endpoints, admin upload handling, and CSV export routes.

## Findings

### 1. High: Default admin credentials and default session secret allow trivial admin compromise

The app falls back to hard-coded admin credentials and a hard-coded session secret when environment variables are missing. Specifically, `ADMIN_EMAIL` defaults to `admin@example.com`, `ADMIN_PASSWORD` defaults to `admin12345`, and `ADMIN_SESSION_SECRET` defaults to `local-dev-session-secret`. The session cookie is then derived deterministically from those values. In any deployment where the env vars are omitted or misconfigured, an attacker can simply log in with the default password or mint the exact expected cookie value.

**Evidence**
- `lib/env.ts`
- `lib/auth.ts`
- `app/api/admin/login/route.ts`

**Impact**
- Full admin takeover becomes possible.
- Because the session token is static for a given credential/secret tuple, compromise is especially easy if defaults remain in place.

**Recommended fix**
- Fail closed in production if `ADMIN_EMAIL`, `ADMIN_PASSWORD`, or `ADMIN_SESSION_SECRET` are unset.
- Replace the deterministic cookie with a random session ID stored server-side, or at minimum sign a structured token with a strong secret that is required to be configured.

### 2. Medium: The captcha answer key is shipped to the client, so bots can bypass it

The captcha challenge object includes both the prompt and the list of accepted answers. That object is imported directly into the client component, and it is also referenced by client-side validation. This means the browser bundle contains the full answer key, so any bot can read the accepted answers and submit them programmatically. The server does validate the answer, but it validates against the same exposed answer set.

**Evidence**
- `lib/contact-captcha.ts`
- `components/contact-seller-form.tsx`
- `lib/validation.ts`
- `app/api/contact-submissions/route.ts`

**Impact**
- The captcha does not meaningfully prevent automated abuse.
- Attackers can spam the contact endpoint while still passing “human verification.”

**Recommended fix**
- Move challenge generation and answer storage fully server-side.
- Send only the prompt and a nonce/challenge ID to the client.
- Validate against server-kept answers or use a standard anti-abuse service/rate limiting.

### 3. Medium: Admin image upload accepts arbitrary file types and sizes, then serves them from `public/`

The admin item route accepts every uploaded `File` with `size > 0`; there is no MIME type validation, extension allowlist, content sniffing, file count limit, or file size limit. The upload helpers then write raw bytes to `public/uploads/...` using a filename extension derived from the user-supplied original filename. That means an attacker who gets admin access can upload arbitrary content such as HTML or SVG and have it served from your origin, which can lead to stored XSS/content spoofing. The lack of size checks also enables disk exhaustion.

**Evidence**
- `app/api/admin/items/route.ts`
- `lib/data/local-store.ts`
- `lib/data/postgres-store.ts`

**Impact**
- Same-origin hosted malicious files can be dangerous, especially HTML/SVG.
- Large uploads can fill disk or exhaust memory because `arrayBuffer()` reads the whole file into memory.

**Recommended fix**
- Restrict uploads to a small allowlist such as JPEG/PNG/WebP.
- Enforce server-side file size and count limits.
- Re-encode images server-side before storage.
- Consider storing uploads outside `public/` and serving them through a controlled handler.

### 4. Medium: CSV exports are vulnerable to spreadsheet formula injection

The CSV export routes escape quotes, but they do not neutralize cells beginning with `=`, `+`, `-`, or `@`. User-controlled values such as `buyerName`, `email`, `message`, `location`, and even item titles are written directly into CSV. If an admin opens an exported file in Excel or Google Sheets, a malicious payload can execute as a formula.

**Evidence**
- `app/api/admin/leads/export/route.ts`
- `app/api/admin/contact-submissions/export/route.ts`
- `app/api/catalogue/export/route.ts`

**Impact**
- Formula injection can trigger outbound requests, data exfiltration, or misleading spreadsheet behavior on the operator’s machine.

**Recommended fix**
- Prefix dangerous cells with a single quote (`'`) or otherwise neutralize leading formula characters before CSV serialization.

## Hardening Notes

### 5. Low/Medium: No evident rate limiting on login or public submission endpoints

The login, lead submission, and contact submission endpoints accept requests directly after schema validation, with no visible throttling, IP-based rate limiting, or abuse controls. That makes credential stuffing and spam significantly easier, especially given the weak captcha design above.

**Evidence**
- `app/api/admin/login/route.ts`
- `app/api/leads/route.ts`
- `app/api/contact-submissions/route.ts`

**Recommended fix**
- Add rate limiting per IP/session for:
  - `/api/admin/login`
  - `/api/leads`
  - `/api/contact-submissions`

## Overall Assessment

The most important issue is the authentication setup: if production ever runs with missing env vars, admin access is effectively exposed. The next biggest problem is the captcha, which is not actually a server-kept challenge. After that, the next priorities should be upload restrictions and CSV injection protections.
